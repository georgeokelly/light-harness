# Light Harness Design (Rule + Hook)

## Why this exists

Many teams already have good rules, but agents still skip them.
This design separates:

- **Rule layer**: policy and contracts (what should happen)
- **Hook layer**: hard backstops (what must not be skipped)

This repository stores source policy in neutral paths (`content/`, `templates/`, `scripts/`).
Runtime adapter files (for example `.cursor/`) are deployment targets, not source-of-truth.

## Rule vs Hook responsibility matrix

| Capability | Rule | Hook |
| --- | --- | --- |
| Intent preflight fields | Define required fields and examples | Block implementation-heavy prompts when preflight is missing |
| Scope guard | Define planned scope semantics | Detect edits outside planned scope and flag |
| Code-change reasoning chain | Define non-skipping step-by-step chain requirement | At stop, warn/block when code changed but chain is missing in record |
| Risk tiering | Define R0-R3 tiers and approval policy | Block high-risk shell commands without approval marker |
| Retry/stop | Define numeric budgets and reset conditions | (Phase 2) block when budget is exhausted |
| Verification ladder | Define V0-V3 criteria and required evidence | Warn on stop when required evidence is missing |
| Local record | Define schema and sections | Warn on stop when record sections are missing |

## Action risk model

- **R0**: read/search only
- **R1**: in-scope edits
- **R2**: runtime commands, scope expansion, unknown side effects
- **R3**: destructive operations, privileged operations, network/install, external side effects

Default policy:

- R0/R1: allow under valid preflight
- R2: explicit approval recommended, may block in strict mode
- R3: explicit approval required

## Verification ladder

- **V0**: disposable spike, minimal verification (must state why V0 is allowed)
- **V1**: static checks (lint/build/test)
- **V2**: runtime smoke check
- **V3**: user/manual verification (only when automation is impossible or unsuitable)

## Local-only record

Record lives under `.agent-memory/records/current/` and should not be committed.

Recommended files:

- `preflight.md`
- `record.md`
- `approvals.md` (optional)
- `scope-violations.log` (generated by hook if needed)
- `hook.log` (generated by hook)
- `_code_change.flag` (generated by hook when code files are edited)

## Rollout

1. Enable rules and strict hooks for missing preflight and high-risk commands.
2. Stabilize templates and record schema.
3. Add retry budget enforcement hooks.
4. Add metrics collection for:
   - missed-constraint rate
   - scope expansion events
   - user-as-default-QA ratio
   - record completeness

## Deployment model

- Source policy: `content/rules/*.md`
- Source hook manifests:
  - `config/hooks/cursor.json`
  - `config/hooks/codex.json`
  - `config/hooks/claude.json`
- Shared merge helper: `scripts/merge_runtime_config.py`
- Local state bootstrap: `scripts/init_local_state.sh`
- Deployment scripts:
  - `scripts/deploy_cursor_runtime.sh`
  - `scripts/deploy_codex_runtime.sh`
  - `scripts/deploy_claude_runtime.sh`
- Local adapter outputs:
  - `.cursor/`
  - `.codex/`
  - `.claude/`

Deploy scripts are merge-aware by default:

- Cursor merges harness-managed hooks into `.cursor/hooks.json`
- Codex merges harness-managed hooks into `.codex/hooks.json` and minimally ensures `.codex/config.toml` enables `codex_hooks`
- Claude Code merges harness-managed hooks into `.claude/settings.local.json` by default, with `--scope project` available for `.claude/settings.json`

For cross-runtime details and limitations, see `docs/adapters.md`.
